### 1.2 数据流

```
用户上传
    │
    ▼
文本提取 (PDF/DOCX/TXT/图片OCR)
    │
    ▼
保存上传记录 (database.add_upload)
    │
    ▼
┌─────────────────────────────────────────────────┐
│              AI Pipeline (3次LLM调用)            │
│  1. 分类判断 → 2. 事实提取 → 3. 卡片建议生成      │

    │
    ▼

    │

    │
    ▼

    │

    │
    ▼

```

### 1.3 AI Pipeline

一次上传内容后，LLM **调用了 3 次**：

| 调用次数 | Prompt | Token 消耗 | 等待时间 |
|:-------:|--------|-----------|---------|
| 第 1 次 | CLASSIFY_PROMPT | ~500-1500 | ~1-3s |
| 第 2 次 | FACT_EXTRACT_PROMPT | ~1000-3000 | ~2-5s |
| 第 3 次 | build_card_suggestions_prompt | ~3000-8000 | ~3-8s |

**总计**：3 次调用，Token 消耗约 4500-12500，等待时间约 6-16s。

### 1.4 当前存在的问题

#### 严重

1. **main.py 过于庞大**（1200+行）：所有逻辑混在一起，难以维护和测试
2. **缺少错误重试机制**：LLM 调用失败直接返回错误，没有重试
3. **缺少请求超时控制**：用户上传大文件或网络慢时可能无响应

#### 一般

4. **重复代码**：`valid_categories` 计算在多个路由中重复
5. **重复 Prompt**：分类体系在多个 Prompt 中重复定义
6. **命名不统一**：`card_text` vs `content` vs `title` 混用
7. **Magic Number**：`50`（标题截断）、`60`（卡片字数限制）等硬编码
8. **缺少缓存机制**：每次聊天都重新检索卡片，无结果缓存

#### 优化建议

9. **数据库连接未复用**：每次操作都新建连接，影响性能
10. **文件提取未缓存**：同一文件重复上传会重复 OCR
11. **全局画像无历史**：只保留最新版本，无法查看变化
12. **缺少卡片搜索能力**：只能按分类浏览，无法关键词搜索
13. **缺少数据统计 API**：前端无法展示分类占比、活跃度等

## 三、AI Pipeline 优化

### 3.1 优化方案：合并第 1 和第 2 次调用

**新流程**：

```
上传内容
    │
    ▼
文件提取（带缓存）
    │
    ▼
┌──────────────────────────────────────────────┐
│ 第1次（合并）: CLASSIFY + FACT_EXTRACT        │
│ 输入: 原始文本                                 │
│ 输出: {"categories": [...], "facts": {...}}   │
└──────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────┐
│ 第2次: build_card_suggestions_prompt          │
│ 输入: 原始文本 + 分类 + 新事实 + 已有卡片       │
│ 输出: {"suggestions": [...]}                  │
└──────────────────────────────────────────────┘
```

### 3.2 优化效果对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| LLM 调用次数 | 3 | 2 |
| Token 消耗 | 4500-12500 | 3000-10000 |
| 等待时间 | 6-16s | 4-11s |

### 3.3 其他优化建议

- **缓存文件提取结果**：使用文件内容 MD5 作为缓存键
- **缓存分类结果**：使用文本 MD5 作为缓存键
- **异步处理**：卡片建议生成后台处理，前端轮询接收

## 四、认知卡片增强

### 4.1 新增字段

| 字段 | 类型 | 用途 | 生成时机 |
|------|------|------|---------|
| `tags` | List[str] | 关键词标签，便于搜索和分类 | 卡片创建时 |
| `keywords` | List[str] | 用于语义匹配的细粒度关键词 | 卡片创建时 |
| `summary` | str | 卡片内容摘要（20-30字） | 卡片创建时 |
| `source_locations` | List[Dict] | 记录事实在原文中的具体位置 | 事实提取时 |
| `confidence` | float | 可信度评分（0-1） | 卡片保存时 |
| `confidence_reason` | str | 可信度评分理由 | 卡片保存时 |

### 4.2 新卡片结构

```python
{
    "id": "uuid_timestamp",
    "card_text": "卡片文本内容",
    "category": "plum",
    "created_at": "2026-07-09T10:00:00",
    "updated_at": "2026-07-09T10:00:00",
    "source_ids": ["upload_id_1"],
    "facts": ["事实1", "事实2"],
    # 新增字段
    "tags": ["内向", "独立思考"],
    "keywords": ["思考", "看书", "深夜"],
    "summary": "喜欢独自思考，常常看书到深夜",
    "source_locations": [
        {
            "source_id": "upload_id_1",
            "start_index": 100,
            "end_index": 150,
            "quote": "我喜欢独自思考问题"
        }
    ],
    "confidence": 0.85,
    "confidence_reason": "所有事实都能在原文中找到明确依据"
}
```

### 4.3 可信度评分规则

| 分数 | 说明 |
|------|------|
| 0.9-1.0 | 所有事实都能在原文中找到逐字对应 |
| 0.7-0.89 | 大部分事实能找到依据，少量有合理推断 |
| 0.5-0.69 | 部分事实能找到依据，存在一定推断 |
| 0.3-0.49 | 事实依据较弱，较多推断 |
| 0-0.29 | 缺乏事实依据，主要是推断 |

### 4.4 兼容性设计

老卡片自动填充默认值：
- `tags`: []
- `keywords`: []
- `summary`: `card_text` 截断前 30 字
- `source_locations`: []
- `confidence`: 0.5
- `confidence_reason`: "历史数据，未评估可信度"

## 六、拾花对话增强

### 6.1 当前问题

1. **关键词匹配过于简单**：只匹配精确关键词，无法处理语义相似的问题
2. **卡片检索策略粗糙**：只取每个分类最新的 1-2 张，没有考虑相关性
3. **LLM 自由发挥空间大**：回答中没有明确引用卡片来源
4. **缺少检索结果缓存**：相同问题重复提问时重复检索和调用

### 6.2 优化方案

#### 6.2.1 增强卡片检索

```python
def search_cards(question: str, top_k: int = 5) -> List[Dict[str, Any]]:
    """增强版卡片检索：关键词 + 标签 + 摘要 + 文本"""
```

#### 6.2.2 组织上下文

```python
def organize_context(cards_list: List[Dict[str, Any]], question: str) -> Dict[str, Any]:
    """按分类分组，生成上下文摘要"""
```

#### 6.2.3 增强回答可解释性

要求回答中明确引用卡片来源：

```python
{
    "answer": "完整回答文本...",
    "referenced_cards": [
        {
            "card_id": "card_id_1",
            "category": "plum",
            "category_name": "梅花",
            "card_text": "卡片内容摘要",
            "evidence": "回答中引用该卡片的具体内容"
        }
    ],
    "categories": ["plum", "peach"]
}
```

#### 6.2.4 添加对话历史

维护最近 5 轮对话历史，构建 Prompt 时加入上下文。

#### 6.2.5 添加结果缓存

使用问题 + 卡片 ID 的 MD5 作为缓存键。

### 6.3 新聊天流程

```
用户提问
    │
    ▼
检查缓存 → 有缓存直接返回
    │
    ▼
关键词匹配分类
    │
    ▼
语义检索卡片（计算相关性分数）
    │
    ▼
组织上下文（按分类分组）
    │
    ▼
构建增强版 Prompt（包含对话历史）
    │
    ▼
调用 LLM 生成回答（要求明确引用卡片）
    │
    ▼
禁用词校验 → 重试（若命中）
    │
    ▼
缓存回答 + 记录对话历史
    │
    ▼
返回回答（包含引用来源）
```

## 七、性能优化

### 7.1 当前性能问题

| 类型 | 问题 | 影响 | 优化方案 |
|------|------|------|---------|
| 重复计算 | `valid_categories` 重复计算 | 中等 | 缓存计算结果 |
| 重复计算 | 分类匹配重复计算 | 高 | 缓存相同文本的分类结果 |
| 重复计算 | 事实提取重复计算 | 高 | 缓存相同文本的事实提取结果 |
| 重复调用 | 聊天回答重复生成 | 高 | 缓存相同问题的回答 |
| 重复调用 | 文件 OCR 重复调用 | 高 | 缓存图片 OCR 结果 |
| 重复调用 | 禁用词重试调用 | 中等 | 在 Prompt 中明确强调规则 |
| 重复解析 | 文件解析重复 | 中等 | 缓存文件解析结果 |

### 7.2 缓存机制设计

```python
class Cache:
    def __init__(self, max_size: int = 1000, ttl_hours: int = 24):
        self.cache: Dict[str, Dict[str, Any]] = {}
        self.max_size = max_size
        self.ttl = timedelta(hours=ttl_hours)
    
    def get(self, key: str) -> Optional[Any]: ...
    def set(self, key: str, value: Any): ...
    def invalidate(self, key_prefix: str): ...
```

### 7.3 缓存应用场景

| 场景 | 缓存键 | 用途 |
|------|--------|------|
| 文件提取 | `file_extract:{md5}` | 避免重复 OCR/解析 |
| 分类+事实提取 | `classify_facts:{md5}` | 避免重复 LLM 调用 |
| 聊天回答 | `chat:{md5(question+card_ids)}` | 避免重复生成 |

### 7.4 缓存失效策略

| 事件 | 失效缓存 |
|------|---------|
| 新增卡片 | `chat:*` |
| 更新卡片 | `chat:*` |
| 删除卡片 | `chat:*` |
| 更新分类体系 | `classify_facts:*` |

### 7.5 其他性能优化

| 优化项 | 收益 | 复杂度 |
|--------|------|--------|
| 数据库 WAL 模式 | 提高并发性能 | 低 |
| 异步 LLM 调用 | 提高响应速度 | 中 |
| 请求限流 | 保护服务稳定 | 低 |

### 8.5 风险评估

#### 高风险
1. **main.py 重构**：涉及核心业务逻辑，修改不当可能导致功能异常
2. **Prompt 合并**：可能影响分类和事实提取的准确性

#### 中风险
3. **缓存机制**：缓存失效策略不当可能导致数据不一致
4. **聊天引擎重构**：语义检索算法可能影响回答质量

#### 低风险
5. **统计模块**：纯计算逻辑，不影响核心功能
6. **前端展示**：仅增加 UI 元素，不影响后端逻辑

### 8.6 开发建议

**建议立即开发**：

1. **Sprint 1（代码质量优化）**：低风险、高价值，为后续所有优化奠定基础
2. **Sprint 2（AI Pipeline 优化）**：中等风险、高收益，直接减少成本和等待时间
3. **Sprint 3（缓存机制）**：低风险、高收益，提升用户体验

这三个 Sprint 可以在 1-2 周内完成，且每个 Sprint 都可以独立测试和回滚。
